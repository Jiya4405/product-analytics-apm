🚨 Root cause detected · Medium–High confidence

The June 2FA rollout is suppressing month-1 retention for new cohorts — established users are unaffected.

At 10,000 signups per cohort (this dataset is a 70-user simulation, scaled here for context), a 33pp retention gap means roughly 3,300 fewer retained users in month 1 alone for any cohort that signs up while this friction point is live.

Supporting evidence
  • ✓ Weekly 2FA challenge failure rate averages 25% since rollout (peak 50%), vs. no 2FA friction possible before June 1 — the feature didn't exist.
  • ✓ Month-1 retention for the May cohort (first cohort to hit the rollout mid-lifecycle) is 32.9%, a 33.3pp drop from the 66.2% Jan–Mar baseline.
  • ✓ The effect holds across all three plan tiers — free (32.4%), pro (34.5%), enterprise (25%, n=4) all show the same May-cohort drop, ruling out a single-segment pricing or product issue.

Month-over-month cohort retention

% of each cohort still active, by months since signup · all plan tiers
View SQL 
-- sql/01_cohort_retention.sql
WITH user_monthly_activity AS (
    SELECT user_id, DATE_TRUNC('month', event_timestamp) AS activity_month
    FROM raw_user_clickstream GROUP BY 1, 2
),
cohort_birth_months AS (
    SELECT user_id, MIN(activity_month) AS cohort_month
    FROM user_monthly_activity GROUP BY 1
)
SELECT c.cohort_month,
       DATEDIFF('month', c.cohort_month, a.activity_month) AS months_since_signup,
       COUNT(DISTINCT a.user_id) AS retained_users
FROM cohort_birth_months c
JOIN user_monthly_activity a ON c.user_id = a.user_id
GROUP BY 1, 2 ORDER BY 1, 2;

Weekly 2FA failure rate

Failed / issued challenges, since Jun 1 rollout
View SQL 
-- sql/03_2fa_incident_detection.sql
SELECT DATE_TRUNC('week', event_timestamp) AS week_start,
       COUNT(CASE WHEN event_type = 'login_2fa_challenge' THEN 1 END) AS challenges_issued,
       COUNT(CASE WHEN event_type = 'login_2fa_failed' THEN 1 END) AS challenges_failed,
       ROUND(COUNT(CASE WHEN event_type = 'login_2fa_failed' THEN 1 END)::NUMERIC
         / NULLIF(COUNT(CASE WHEN event_type IN
             ('login_2fa_challenge','login_2fa_failed') THEN 1 END), 0)::NUMERIC * 100, 2
       ) AS two_fa_failure_rate_pct
FROM raw_user_clickstream
GROUP BY 1 ORDER BY 1;

Pipeline & engineering trade-offs

How a row of clickstream data becomes a Decision Center brief
Raw CSV→ Python (generate_clickstream_data.py)→ DuckDB / SQL (sql/*.sql)→ Weekly aggregation→ Claude API→ This dashboard
Why aggregate before Claude?

Privacy — only weekly counts leave the system, never a user_id or raw row. Cost — a 9-row summary table is a fraction of the tokens 9,660 raw events would cost. Latency — one small request beats streaming thousands of rows through the model.

What if the API is down?

ai_analytics_pipeline.py catches the API error and falls back to printing the same prompt and metrics table for manual review — the PM never loses the underlying data, only the auto-summary.

What if Claude hallucinates?

Every claim in the Decision Center above is traceable to a SQL query you can expand and re-run (see "View SQL" on each chart) — the model explains the numbers, it doesn't originate them.
